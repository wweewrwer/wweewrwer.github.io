实现操作系统的主要原因：隔离性和分时复用（multiplexing)。实现隔离性的方法：User/kernel mode和虚拟内存。

Kernel和User mode,用来隔离操作系统内核和用户应用程序。user mode只能执行普通权限的指令，如寄存器加减，而kernel mode则可以执行一些直接操纵硬件的指令和设置保护的指令，如设置page table寄存器，关闭时钟中断。应用程序要想转入内核，需要一个专门的指令，ECALL，传入一个数字表示应用程序想调用的System Call。在XV6中，每一个应用程序发起的系统调用都会调用syscall函数，该函数会检查ECALL参数，判断有没有资格去执行对应函数。

System calls，应用程序转移到内核执行的基本方法。

宏内核：整个操作系统代码都运行在kernel mode。缺点：bug多，任何一个操作系统的bug都有可能成为漏洞。优点：将如文件系统，虚拟内存等子模块都集成在了一起，性能好。
微内核：希望kernel mode运行尽可能少的代码。大部分操作系统运行在内核外。优点：简单，更少的bug。缺点：反复跳转带来的性能损耗

Q: 三级页表为什么省内存
A: 因为第一级页表中空的PTE可以不申请其第二级和第三级的内存，等到其实际使用了再申请，故省内存。

Q: 三级页表和那个Kernel address space有什么关系
A: 三级页表是存的方式，kernel address space是具体内容，就好比kernel address space是一杯水，三级页表和一级页表则分别是一个方杯子和一个圆杯子，用哪个杯子来装水和水本身是什么组成的是没有关系的，它们的都可以装水。即 三级页表和kernel address space的内容是没有关系的

Q：那又有process address space又有kernel address space它们分属不同的页表，申请实际的物理内存不会冲突吗
A：申请到的内存由底层决定，好比一个停车场同时有10个人来申请车位，停车场管理员依次分配管理即可。产生这个问题的原因是各个知识点没有理清，比较混沌，才会产生这个问题

Q：那用户栈和内核栈又是什么？有什么区别？它们的大小是固定的吗？
A: 先回答最后一个问题，在实际的操作系统中，可能不是固定的，比如可能动态增长，但是，在XV6中，见3.6的最后“If the user stack overflows and the process tries to use an address below the stack, the hardware will generate a page-fault exception because the mapping is not valid. A real-world operating system might instead automatically allocate more memory for the user stack when it overflows.”，即是固定的
然后要了解这两类栈，首先得回想起栈是干什么的，可以结合以前自己学c/c++语言的知识，栈上面存放着一些局部变量和函数参数。进程通过系统调用进入内核后所使用的栈即为内核栈，而平时使用的则是用户空间的栈。
Q：虚拟地址到物理地址是系统做的吗？
A：不是，是由硬件完成的，只有处理缺页需要硬件和操作系统合作完成。cpu中有一个页表地址寄存器指向当前页表，运行时虚拟地址会自动翻译。

LEC6

这里是讲系统调用，不是中断，但是都是用的相同的机制
trap机制：即用户空间和内核空间的切换
具体发生的情况为：程序执行系统调用，程序出现类似page fault、运算时除以0的错误，或者一个设备触发了中断使得当前程序运行需要响应内核设备驱动
非常重要，该机制要尽可能简单
切换的时候要关注的：sp（堆栈寄存器），pc（程序计数器），表示当前mode的标志位（表明当前在supervisor mode还是user mode）。控制CPU工作方式的寄存器（supervisor address translation and protection)：satp（包含了指向page table的物理内存地址）。STVEC寄存器：指向内核中处理trap的指令的起始地址。SEPC：trap过程中保存程序计数器的值。
中断过程：1.首先，保存32个用户寄存器 2.改变mode为内核模式 3.修改satp指向内核的page table。 4.将堆栈寄存器指向位于内核的一个地址，需要一个堆栈来调用内核的c函数 5.跳入内核的c代码
trap过程不能依赖用户空间的东西，因为安全和隔离。supervisor mode能做的事：1.读写控制寄存器（如satp,stvec,sepc等） 2.可以使用PTE_U为0的页表
注：supervisor mode也不能任意读写物理地址，也要通过page table访问（这样好像理解了每个进程单独内核页表的作用，这样每个进程就没办法操作其它的进程的内存）
调用过程：uservec-trampoline然后是uservec之后usertrap随后执行syscall随后执行对应的sys调用
trap.c usertrapret完成部分返回用户空间的工作![img](/img/image (183).png)

进入内核的流程：

  1. ecall:

  • 改为supervisor mode。

  • 将程序计数器的值保存在sepc寄存器。

  • 将程序计数器改动来到trampoline page（该page包含了内核trap的处理代码）

  1. 进入uservec函数，开始执行trampoline page处的指令，先csrrw交换a0和sscratch（sscratch会事先存好trapframe page的虚拟地址）的数据，这样腾出来a0寄存器（因为要保护原有寄存器的内存），同时得到了指向trapframe page的内存，可以去保存其它数据。内核要首先设置好stvec寄存器，使它指向trampoline page的起始位置，然后ecall后就按该指令跳转。注意trampoline page不设置pte_u，这样用户空间是不能写它的。此时执行trapframe page后面的指令，就是保存其它的寄存器。随后加载内核的stack pointer。然后向tp寄存器写入数据（这个寄存器保存cpu核的编号）。之后satp寄存器。注意trampoline page在user page table和kernel page table的映射都是一样的（名字来源：就像弹跳了一下，从用户空间到内核空间，故叫蹦床界面）最后跳转到内核的c代码
  2. 此时从usertrap函数开始。位于trap.c。设置stvec寄存器指向内核空间trap处理代码的位置。保存sepc寄存器到该进程关联的内存（因为不然可能再次trap，然后导致保存的被覆盖）这里保存到trapframe中，防止被覆盖。利用scause寄存器找出现在触发trap的原因。之后保存的sepc+4，因为要恢复到ecall的下一条指令。此时打开中断开关，允许中断的出现
  3. 接着调用syscall函数，该函数定义在syscall.c。根据系统调用的编号查找相应的系统调用函数
  4. 之后回到trap.c中的usertrap函数，检查用户进程是否被杀掉，被杀掉就不恢复了。
  5. 接着调用usertrapret函数。首先关闭中断，此时将stvec指向用户空间的trap处，接下来填入trapframe，存储kernel page table的指针，用户进程的kernel stack,usertrap函数的指针，tp寄存器中读取的当前cpu核的编号，并存储在trapframe中。接着设置sstatus寄存器。接着sret，设置sepc寄存器的值
  6. 接下来进入userret函数，切换page table,清空页表缓存，交换a0和ssratch，然后恢复其它寄存器，然后再交换sscratch寄存器和a0的值，然后执行sret，回到user mode,sepc拷贝到pc,重新打开中断。然后回到用户空间中。

ecall之后还有保存寄存器，切换到kernel page table，创建内核栈，跳转到内核中c代码的合理位置一系列的工作。ecall并不完成这些，而是软件完成，有些机器会硬件完成，但risc-v不会，因为risc-v想要为软件和操作系统的程序员提供最大的灵活性，比如可能某些操作系统可以不切换page table的前提下执行部分系统调用，如果ecall都做了，这就不行了，增加了开销

LEC8

page fault:
要处理page fault要知道：
1.出错的虚拟地址，保存在stval寄存器
2.出错的原因，scause（Supervisor cause）寄存器储存出错的原因
3.触发page fault指令的地址，sepc（Supervisor Exception Program Counter）寄存器
Lazy allocation:应用程序申请内存的时候，不实际增加内存，当它用到的时候，触发page fault,然后再增加，避免应用程序申请了不用的内存而产生浪费
实现的基本思路：首先要注意有两个地方要改（而不是一个），第一是前面说的用到未分配内存要增加，第二个是释放内存遇到还未分配的要跳过
Zero fill on demand:我们很多时候会拥有大量内容全是0的page，因此可以只分配一个page，该page全是0，然后将虚拟地址上全0的page都map到该物理page上，这样至少在程序启动时能节省大量的物理内存分配。然后将该page标记为只读，一但尝试写就会page fault，然后新建一个page，更新为可读可写，更新mapping关系，将其内容设置为0，并重新执行指令
注意：一个trap的代价是相当高的，可以看trap的处理代码，里面至少有上百个store指令用来储存寄存器等，还有一些其它的开销
copy on write fork: 当fork接exec时，fork会拷贝原来的地址空间，而exec会丢掉原来的地址空间，这样就产生了浪费。这里的优化就是当发生拷贝时，不实际拷贝，而是指向同一个地方，当要写时再拷贝。要注意很多细节（例如何判断是copy on write fork，而不是非法写,一个进程释放的时候怎么办之类）
Demand paging: 不立即将程序的text,data加载进内存，等到需要的时候再加载。同时在内存不够的时候，可以撤回已经分发的page，可以通过LRU来实现，同时额外标记有无被修改，优先撤回一定时间内没有被修改的
Memory mapped files:将完整或者部分文件加载到内存中，这样就可以通过内存地址相关的load和store来操纵文件，然后通过dirty标志位写回。现代操作系统会提供一个叫mmap的系统调用，一般该调用也是lazy的，需要的时候再加载

LEC 9

电脑小知识：计算机大部分内存都会在使用中，但是大部分内存并没有被应用程序所使用，而是被buff/cache用掉了。这在一个操作系统中还挺常见的，因为我们不想让物理内存就在那闲置着，我们想让物理内存被用起来，所以这里大块的内存被用作buff/cache。（怪不得我电脑每次16G内存都占用这么高，搞得我一直在想难道8G内存的是怎么用的电脑）
中断这里要理解一下哪些是硬件实现，哪些是软件实现，实际上，硬件可以就理解成一个接口，硬件可以提供接口当前是否允许中断，然后软件来操作，同时发生中断了转到中断程序这个（在这个实验里）也是硬件实现的（网上说好像也可以软件)
系统调用与中断的区别：

  1. asynchronous。当硬件生成中断时，Interrupt handler与当前运行的进程在CPU上没有任何关联。但如果是系统调用的话，系统调用发生在运行进程的context下。

  2. concurrency。对于中断来说，CPU和生成中断的设备是并行的在运行。网卡自己独立的处理来自网络的packet，然后在某个时间点产生中断，但是同时，CPU也在运行。所以我们在CPU和设备之间是真正的并行的，我们必须管理这里的并行。

  3. program device。LEC9主要关注外部设备，例如网卡，UART，而这些设备需要被编程。每个设备都有一个编程手册，就像RISC-V有一个包含了指令和寄存器的手册一样。设备的编程手册包含了它有什么样的寄存器，它能执行什么样的操作，在读写控制寄存器的时候，设备会如何响应。不过通常来说，设备的手册不如RISC-V的手册清晰，这会使得对于设备的编程会更加复杂。

注意，目前是看外部设备的中断，实际上还有定时器和软件中断。
各类设备会映射到内核内存地址的某处，通过向相应的设备地址执行load/store指令就可以对该设备进行编程
处理器通过platform level interrupt control，即PLIC来处理设备中断，具体流程为PLIC会通知当前有一个待处理的中断->其中一个CPU核会Claim接收中断，这样PLIC就不会把中断发给其他的CPU处理->CPU核处理完中断之后，CPU会通知PLIC->PLIC将不再保存中断的信息。注意PLIC只是分发中断，内核可以对中断优先级进行编程等，这里非常灵活。
管理设备的代码称为驱动，所有的驱动都在内核中。大部分驱动可以分为两部分，bottom/top。bottom通常是interrupt handler，它不运行在任何特定的进程的context中，只是处理中断。top是用户进程和内核其它部分调用的接口。通常情况下驱动有一些队列，top和bottom部分都会从队列中读写数据
RISC-V与中断相关的寄存器：

  • SIE（Supervisor Interrupt Enable）寄存器。这个寄存器中有一个bit（E）专门针对例如UART的外部设备的中断；有一个bit（S）专门针对软件中断，软件中断可能由一个CPU核触发给另一个CPU核；还有一个bit（T）专门针对定时器中断。
  • SSTATUS（Supervisor Status）寄存器。这个寄存器中有一个bit来打开或者关闭中断。这个寄存器中有一个bit来打开或者关闭中断。每一个CPU核都有独立的SIE和SSTATUS寄存器，除了通过SIE寄存器来单独控制特定的中断，还可以通过SSTATUS寄存器中的一个bit来控制所有的中断。
  • SIP（Supervisor Interrupt Pending）寄存器。当发生中断时，处理器可以通过查看这个寄存器知道当前是什么类型的中断。
  • SCAUSE寄存器，表明当前状态的原因是中断。
  • STVEC寄存器，保存当trap，page fault或者中断发生时，CPU运行的用户程序的程序计数器，这样才能在稍后恢复程序的运行。
    后两个寄存器在中断处理流程中与之前系统调用的工作方式基本一样。
    流程：

  1. start.c中的start函数，所有的中断都设置在Supervisor mode，之后设置SIE寄存器来接收External，软件和定时器中断，之后初始化定时器

  2. 接下来这里只考虑external中断。进入main函数，依次进行init，例如这里进行了uartinit

  3. 接下来设置PLIC，使中断能被CPU感知。具体方法为在main函数里，调用plicinit函数设置接收哪些中断

  4. 然后plicinithart函数，每个核均需调用plicinithart函数表明对哪些外设中断感兴趣，并设置中断优先级。

  5. 调用scheduler函数，该函数首先调用intr_on使CPU能接收中断，然后开始运行进程

UART驱动的top部分：创建console设备，文件描述符0，之后通过dup创建stdout和stderr。所以这里文件描述符0,1,2都是表示console（在unix中，设备由文件表示）。然后当要往console写时，前面部分就和平常一样，fprintf呀之类的，然后进入sys_write系统调用，调用filewrite函数，判断文件描述符的类型，如果是设备，则执行相应的write函数，在里面调用uartputc函数将字符写入到UART设备，然后通过uartstart通知设备执行操作，
UART驱动的bottom部分：当向console输出字符时发生了中断：1.清除SIE寄存器相应的bit，防止被其它中断干扰。2.设置sepc为当前程序计数器。3.保存当前的mode。4.将mode设置为supervisor mode。5.最后将程序计数器的值设置为STVEC的值。（注：STVEC会指向uservec函数，见前面系统调用部分，这一部分代码之前处理了系统调用和Page fault,现在来处理中断）6.然后进入trap.c中的devintr函数，首先通过SCAUSE寄存器判断当前中断是否是来自外设的中断，如果是，再调用plic_claim函数来获取中断。该函数中CPU核会告知PLIC要处理中断，之后PLIC_SCLAIM将中断号返回，之后调用uartintr函数，会从UART的接受寄存器中读取数据然后传递给consoleintr函数，之后代码运行到uartstart函数，该函数会将Shell存储在buffer中的任意字符送出
与Interrupt相关的并发：1.设备与CPU是并行运行的。 2.中断会停止当前运行的程序。3.驱动的top和bottom部分是并行运行的。
这里生产者和消费者解耦是通过一个buffer进行的，这样他们可以同时运行。
中断的发展：现在除了interrupt还有polling(轮询)，即CPU一直不停的轮询设备，直到设备有了数据。对于快设备，经常interrupt的开销很高，因而可以采用polling。一些驱动会在polling和interrupt之间动态的切换

LEC10

race condition(竞争条件)：由于两个或者多个进程竞争使用不能被同时访问的资源，使得这些进程有可能因为时间上推进的先后原因而出现问题
锁的作用：避免丢失更新。打包多个操作，使它们具有原子性。维护共享数据结构的不变性。
deadly embrace(死锁)：这里的解决方法是对锁进行排序，所有的操作必须按相同的顺序获取锁。（例如你要对d1,d2加锁，必须都先获取d1锁，然后才能获取d2锁）不过在设计一个操作系统的时候，定义一个全局的锁的顺序会有些问题。如果一个模块m1中方法g调用了另一个模块m2中的方法f，那么m1中的方法g需要知道m2的方法f使用了哪些锁。因为如果m2使用了一些锁，那么m1的方法g必须集合f和g中的锁，并形成一个全局的锁的排序。这意味着在m2中的锁必须对m1可见，这样m1才能以恰当的方法调用m2。但是这样又违背了代码抽象的原则。在完美的情况下，代码抽象要求m1完全不知道m2是如何实现的。但是不幸的是，具体实现中，m2内部的锁需要泄露给m1，这样m1才能完成全局锁排序。所以当你设计一些更大的系统时，锁使得代码的模块化更加的复杂了。
通常来说，这里开发的流程是：先以coarse-grained lock（注，也就是大锁）开始。再对程序进行测试，来看一下程序是否能使用多核。如果可以的话，那么工作就结束了，你对于锁的设计足够好了；如果不可以的话，那意味着锁存在竞争，多个进程会尝试获取同一个锁，因此它们将会序列化的执行，性能也上不去，之后你就需要重构程序。
锁的实现：主要是通过利用硬件的方式，有很多种，通过硬件锁确保软件锁不会被同时获得。注意，一条普通硬件指令不一定是原子操作，例如store指令会取决于具体的实现，可能不是原子的
这里锁要处理两类并发：一类是不同CPU之间的并发，一类是相同CPU上中断和普通程序之间的并发。
注意编译器或者处理器可能通过重排指令来获得更好的性能，但是对于并发执行，一些重排可能会出现问题，因此，为了告诉编译器和硬件不要这样做，我们需要使用memory fence或者叫做synchronize指令，来确定指令的移动范围。对于synchronize指令，任何在它之前的load/store指令，都不能移动到它之后。锁的acquire和release函数都包含了synchronize指令。

LEC11

多线程的原因：1.让计算机支持同一时间不是只执行一个任务，或者让计算机执行分时复用的任务 2.让程序的结构变得简单 3.使用多线程可以通过并行运算，在拥有多核CPU的计算机上获得更快的处理速度
线程可以认为是一种在有多个任务时简化编程的抽象。一个线程可以认为是串行执行代码的单元。在这里，认为线程就是单个串行执行代码的单元，它只占用一个CPU并且以普通的方式一个接一个的执行指令。
线程的状态：1.程序计数器（Program Counter），它表示当前线程执行指令的位置。2.保存变量的寄存器。3.程序的Stack。通常来说每个线程都有属于自己的Stack，Stack记录了函数调用的记录，并反映了当前线程的执行点。
多线程的策略：1.多个核上使用多个cpu 2.（主要）一个CPU在多个线程之间来回切换。
xv6每个进程只有一个线程（注：单进程的多用户线程共享内存，进程之间用户线程不共享内存，但是进程之间内核线程是共享内存）。同时，除了进程之外，还有一些其他的方式可以支持在一台计算机上交织的运行多个任务，如event-driven programming或者state machine。在所有的支持多任务的方法中，线程技术并不是非常有效的方法，但是线程通常是最方便，对程序员最友好的，并且可以用来支持大量不同任务的方法。
实现内核中线程系统的挑战：1.如何实现线程间的切换。2.决定线程的哪些信息是必须保存的，并且在哪保存它们。3.如何处理运算密集型线程（compute bound thread）（对于线程切换，很多直观的实现是由线程自己自愿的保存自己的状态，再让其他的线程运行。但是如果我们有一些程序正在执行一些可能要花费数小时的长时间计算任务，这样的线程并不能自愿的出让CPU给其他的线程运行。所以这里需要能从长时间运行的运算密集型线程撤回对于CPU的控制，将其放置于一边，稍后再运行它）
处理3的方式：利用定时器中断。这样的流程叫pre-emptive scheduling，即用户代码本身没有出让CPU，定时器中断仍然会将CPU的控制权拿走，并出让给线程调度器。相对的叫voluntary scheduling。
在调度时，要能区分以下几类线程：1.当前在CPU上运行的线程(对应xv6的RUNNING线程状态），2.一旦CPU有空闲时间就想要运行在CPU上的线程（RUNABLE） 3.以及不想运行在CPU上的线程，因为这些线程可能在等待I/O或者其他事件（SLEEPING)
从一个进程切换到另一个进程的流程： （注：实际上XV6切换线程的绝大部分场景都不是因为定时器中断）

  1. 从第一个用户进程接入到内核中，保存用户进程的状态并运行第一个用户进程的内核线程。具体实现为内核线程调用swtch函数，将用户寄存器存在trapframe中，内核线程的寄存器存在context中。

  2. 再从第一个用户进程的内核线程切换到第二个用户进程的内核线程。具体做法为首先内核线程切换到CPU对应的调度器线程。然后执行schedulder函数，找到下一个要执行的进程p2，调用swtch函数，然后将p2从之前的swtch中恢复，然后返回到进程P2所在的系统调用或者中断处理程序中

  3. 之后，第二个用户进程的内核线程暂停自己，并恢复第二个用户进程的用户寄存器。这里就和之前LEC6是差不多的

  4. 最后返回到第二个用户进程继续执行。

每个CPU都有一个完全不同的调度器线程。调度器线程也是一种内核线程，它也有自己的context对象。任何运行在CPU1上的进程，当它决定出让CPU，它都会切换到CPU1对应的调度器线程，并由调度器线程切换到下一个进程。
注：context swithcing通常指一个线程切换到另一个线程，有时也指从一个用户进程切换到另一个用户进程的完整过程，偶尔也指从用户空间和内核空间之间的切换，LEC10指一个内核线程和调度器线程之间的切换
具体代码分析：yield函数：当前进程会出让CPU并让另一个进程运行。sched函数：做合理性检查。然后swtch函数：保存当前的内核线程的寄存器，同时将要切换的线程的寄存器恢复到CPU的寄存器中，同时ra(return address)寄存器保存为scheduler函数
Linux多线程：几乎可以认为Linux中的每个线程都是一个完整的进程。Linux中，我们平常说一个进程中的多个线程，本质上是共享同一块内存的多个独立进程。所以Linux中一个进程的多个线程仍然是通过一个内存地址空间执行代码。
个人感悟：所以这里的切换进程，即是切换上下文（context）然后继续执行就可以了，上下文变了相当于就是变了一个进程了

LEC13

在sched函数中，进程p不能持有除p->lock外的其它锁，不然可能导致死锁，例如假如只有一个CPU核，进程切换了另一个进程，而另一个进程想要使用该锁时就可能产生死锁（假如该锁是自旋锁的话）（这里感觉有一点点奇奇怪怪，可能是因为是自旋锁的原因，XV6里面在获取锁的过程中是会关中断的，就是该进程在等待锁的时候是无法切换的)（不过它这里保证了切换的时候不加锁，那么相当于进程在等的话意外着当前有进程正在用，那么很快就会结束，所以也不是很影响性能）
coordination：出让CPU，直到等待的事件发生再恢复执行（例如等待键盘输入）。
sleep和wakeup机制：在等待的过程调用sleep函数出让cpu，然后满足条件的地方调用wakeup函数。(这里sleep的方式为出让CPU，并设置状态为SLEEPING，而wakeup并不是拿回CPU，而是将其设置为RUNABLE，这样上一讲里的上下文切换时就可以切换到这个地方)由于上述切进程我们不能持有锁的原因，故我们的sleep和wakeup需要有一个参数放锁。用于进入sleep之后解锁。这里sleep需要保证原子性的将进程设置为SLEEPING状态并同时释放锁。这里的sleep和wakeup和条件变量很像，也是要放循环里，不过这里不是因为硬件虚假唤醒的原因，而是比如有两个进程都在sleep等待往硬件中写，这时硬件准备好了，两个都会被唤醒，但是应该只有一个能写，故要while。
wakeup实现：查看进程表单查找要唤醒的线程，设置该进程的状态为RUNNABLE。注意查找比对的时候要先获得进程锁
sleep实现：先获得进程锁，然后解开传入的锁，并设置状态，由于有进程锁，所以现在wakeup不能解开，这样就保证了原子性
进程退出的问题：
1.当程序主动调用exit时，我们需要一种方法让线程能释放最后几个对于运行代码来说关键的资源。
2.我们不能直接单方面的kill或者exit一个进程，因为：该线程可能正在另一个CPU核上运行，并使用着自己的栈；也可能该线程正在内核中持有了锁；也可能该线程正在更新一个复杂的内核数据，如果我们直接就把线程杀掉了，我们可能在线程完成更新复杂的内核数据过程中就把线程杀掉了。
exit进程的实现：释放进程的内存和page table，关闭已经打开的文件，同时，若为子进程，需要将父进程从wait中唤醒，若为父进程，需要将子进程设置为init进程的子进程。设置状态为ZOMBIE,因为资源还没被完全释放，进入sched函数进到调度器线程。(注意，具体实现的细节甚至要考虑子进程和父进程同时退出的这些情况)
wait调用：里面包括一个大的循环，当一个进程调用了wait系统调用，它会扫描进程表单，找到父进程是自己且状态是ZOMBIE的进程，然后调用freeproc函数释放这些进程的资源并将其设置为unused。（注意在unix里，每一个子进程退出都需要一个wait，init进程的工作就是在一个循环里不停的调用wait)
kill调用：直接关掉一个进程是有风险的，在xv6和其它unix系统中kill实际干的事很少，它只是找到目标进程，然后将该结构体的killed标志位设为1，同时如果是sleeping状态，设置为runnable（这是防止比如在等控制台输入这种情况，不然要一直等下去才能退出，同时要支持这种情况，需要在sleep的while循环中添加if killed退出的语句，这样开始执行就可以退出了，如果必须等待的话，就不加这个判断，这样唤醒之后又会不满足while条件再次sleep），然后等到其到达安全的位置，再主动调用exit系统调用。所以，kill系统调用并不是真正的立即停止进程的运行，它更像是这样：如果进程在用户空间，那么下一次它执行系统调用它就会退出，又或者目标进程正在执行用户代码，当时下一次定时器中断或者其他中断触发了，进程才会退出。所以从一个进程调用kill，到另一个进程真正退出，中间可能有很明显的延时。
注意：init进程是不退出的，在一些发生错误的情况是会退出，这种情况下系统会崩溃
